Every bug bounty hunter and red team operator knows the drill: before you can find vulnerabilities, you need to map the terrain. That means hours of manual recon — running tool after tool, copy-pasting outputs, cross-referencing CVE databases, and eventually stitching a picture together from a dozen text files.

I got tired of that. So I built RTeamF — an end-to-end automated reconnaissance framework that takes a single domain as input and handles everything else: subdomain discovery, liveness checks, visual recon, JavaScript crawling, port scanning, tech-stack fingerprinting, CVE mapping, and risk scoring. Then it stores the results in AWS S3 and MongoDB.

This post walks through the architecture, the key design decisions, and the code that makes it work.

The Architecture

Nine tools. One pipeline. Zero manual steps.

The core insight behind RTeamF is simple: every major recon tool accepts stdin or a file as input and produces stdout or file output. That means they can be chained. Python's subprocess module is the glue — it orchestrates each tool, captures its output, and feeds it downstream.

1

Subdomain Discovery

Three tools run in parallel — Sublist3r, Subfinder, and Assetfinder — to maximize subdomain coverage. Results are deduplicated and filtered against a blacklist of CDN noise like awstrack.me and bit.ly.

Two parallel vulnerability sources: Nuclei scans for live misconfigurations and exploits using critical/high/medium/low severity templates. Simultaneously, the NIST NVD REST API is queried with each discovered CPE to map the exact tech stack to known CVEs, with Exploit-DB IDs fetched for each hit.

Deep Dive #1

Chaining tools with Python subprocess

The entire orchestration layer is built on Python's subprocess module. Each tool is invoked, its output captured, cleaned, and passed along. Here's a simplified view of how subdomain discovery feeds into liveness checking:

import subprocess # Step 1: collect raw subdomains from three tools def enumerate_subdomains(domain): raw = set() for tool_cmd in [ ["subfinder", "-d", domain, "-silent"], ["assetfinder", "--subs-only", domain], ["sublist3r", "-d", domain, "-o", "/dev/stdout"], ]: result = subprocess.run(tool_cmd, capture_output=True, text=True) raw.update(result.stdout.splitlines()) return filter_blacklist(raw) # Step 2: pipe subdomains into httprobe for liveness def check_liveness(subdomains): joined = "\n".join(subdomains) proc = subprocess.run( ["httprobe"], input=joined, capture_output=True, text=True ) return proc.stdout.splitlines() # only live URLs returned
The key design principle: every function consumes the previous function's output. There's no shared mutable state — just clean data flowing downstream, stage by stage.
Deep Dive #2

From tech stack to CVEs — the CPE bridge

This is the most technically interesting part of RTeamF. Once Wappalyzer identifies the technologies running on a target — say, WordPress 6.2, Apache 2.4.51, PHP 8.1 — those are normalized into CPE strings. CPE is a standardized naming scheme used by NIST to catalog software.

With a CPE in hand, RTeamF hits the NIST NVD REST API directly:

import requests def query_nvd_for_cves(cpe_string): url = "https://services.nvd.nist.gov/rest/json/cves/2.0" params = { "cpeName": cpe_string, "resultsPerPage": 50 } resp = requests.get(url, params=params) data = resp.json() cves = [] for item in data.get("vulnerabilities", []): cve = item["cve"] cves.append({ "id": cve["id"], "severity": cve["metrics"].get("cvssMetricV31", [{}])[0] .get("cvssData", {}).get("baseSeverity", "UNKNOWN"), "description": cve["descriptions"][0]["value"] }) return cves

Each returned CVE ID is then passed to cve_searchsploit to retrieve any corresponding Exploit-DB entries. If a working public exploit exists for a CVE found on your target, RTeamF will surface it automatically.

Deep Dive #3

The risk scoring algorithm

Raw CVE counts and Nuclei findings are useful, but hard to act on. RTeamF's API.py module calculates a unified security posture score out of 100 using a weighted severity model:

severity weight table

×5 critical
×4 high
×2 medium
×1 low / info

The weighted sum is normalized against a theoretical maximum and inverted — a score of 100 means zero findings, while lower scores indicate greater exposure. Thresholds map to labeled risk tiers: Safe → Moderate → High Risk.

Deep Dive #4

Containerizing the entire stack

One of the biggest headaches in security tooling is dependency management. Nuclei, nmap, httprobe, Wappalyzer, wig — each has its own quirks, system dependencies, and version requirements. On a fresh machine, getting all of them working reliably is a multi-hour exercise.

RTeamF solves this by containerizing everything on kalilinux/kali-rolling:

# Dockerfile (abbreviated) FROM kalilinux/kali-rolling # OS-level security tools RUN apt-get install -y \ nmap subfinder nuclei httprobe \ chromium-driver python3-pip # Python layer with custom Wappalyzer + wig overrides COPY requirements.txt . RUN pip3 install -r requirements.txt COPY custom_require/ /overrides/ ENTRYPOINT ["python3", "code.py"]

The "custom require overrides" are worth calling out. Both Wappalyzer and wig have upstream maintenance issues — their PyPI packages lag behind or break on newer Python versions. Rather than fork them publicly, RTeamF vendors patched copies that are layered in during the Docker build. This keeps the container self-contained and reproducible.

The practical payoff: you can spin up a distributed scan by launching multiple containers in parallel, each targeting a different domain, with results aggregating into the same MongoDB cluster. No environment inconsistencies. No "works on my machine."

Storage Layer

Two-tier persistence: S3 for raw data, MongoDB for intelligence

After a scan completes, RTeamF performs two separate writes:

AWS S3

Full scan directories — screenshots, raw tool outputs, nmap XML — uploaded recursively for long-term storage and audit.

MongoDB

Structured JSON intelligence — per-subdomain records with ports, stack, CVE counts, exploit IDs, and posture score — inserted into cloudRTF cluster.

JSON Schema

Each record maps subdomain → open ports → tech stack → JS links → CVE count → exploit count → risk score. Queryable instantly.

This two-tier approach separates concerns cleanly: S3 is the audit trail and raw evidence store, while MongoDB is the operational intelligence layer you'd query for dashboards, alerting, or downstream workflows.

Lessons Learned

What I'd do differently

Rate limiting the NVD API

NIST's NVD API is public but rate-limited. On large scans with many unique CPEs, you'll hit the threshold quickly. The current implementation uses a simple retry with backoff, but a proper token-bucket rate limiter and an API key (which raises the limit significantly) would be the production-grade solution.

Async tool orchestration

The current pipeline is largely sequential. Many stages — nmap, Wappalyzer, Nuclei — are independent of each other given the same input (the live subdomain list). Migrating to asyncio or a lightweight task queue like Celery would cut total scan time significantly on large target surfaces.

False positive filtering

Nuclei templates, particularly at the info and low severity level, produce noise. A confidence scoring layer — weighting findings by template quality, corroboration across tools, and historical false-positive rates — would make the output more actionable.

RTeamF started as a personal productivity tool for cutting recon time during engagements. It's evolved into something more interesting: a demonstration that the gap between raw security tooling and structured intelligence can be closed with a few hundred lines of Python, a Dockerfile, and some thoughtful orchestration. The tools existed. The APIs existed. The missing piece was the glue — and now it's automated.
